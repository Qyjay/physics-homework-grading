## 一、商业模式画布（Business Model Canvas）

### 1. 客户群体（Customer Segments）

本系统面向三类核心用户群体：

- **高中物理教师（主要用户）：** 承担繁重批改任务的一线教师，尤其是大班教学（40-60 人/班）的教师，每天需批改 120-200 份作业，急需提升效率。
- **高中学生（终端受益者）：** 希望获得即时、详细反馈的学生，尤其是缺乏一对一辅导资源的普通学校学生。
- **学校管理层 / 教务部门（决策采购者）：** 关注教学质量提升和教师减负的学校管理者，是付费决策方。

### 2. 价值主张（Value Propositions）

- **对教师：** 将单题批改时间从 3-5 分钟缩短至 15 秒以内，释放教师精力用于教学设计和个别辅导；提供统一的评分标准，消除主观偏差。
- **对学生：** 作业提交后即时获得逐步反馈（而非等待 1-3 天），包括具体错误定位、错因分析和改正建议，抓住最佳纠错时机。
- **对学校：** 提供过程性评价数据（非仅最终分数），帮助教研组精准定位教学薄弱环节；缓解师资不足问题，尤其利好乡镇学校。
- **差异化：** 市面搜题产品（作业帮、猿辅导）侧重「搜答案」，本系统侧重「改过程」——对推导步骤逐步检查，具备受力分析图理解能力，是目前市场空白。

### 3. 渠道（Channels）

- **Web 端：** 教师通过浏览器访问系统，上传作业照片或扫描件，查看批改结果和班级报告。这是 MVP 阶段的主要渠道。
- **移动端（后续）：** 学生通过手机拍照直接提交，实现即拍即改。
- **触达渠道：** 通过教育行业展会、物理教师社群（QQ 群 / 微信群）、教研活动推广；与教育局或学校合作试点。

### 4. 客户关系（Customer Relationships）

- **自助服务：** 教师自行注册、上传作业、查看结果，零学习成本。
- **社区驱动：** 建立物理教师用户社群，收集反馈、共享标准答案模板，形成内容共建生态。
- **校级对接：** 对批量采购的学校提供专属部署支持和培训。

### 5. 收入来源（Revenue Streams）

- **免费增值（Freemium）：** 基础批改功能免费（每月限额），吸引教师试用。
- **订阅制（SaaS）：** 教师个人月度/年度订阅（不限量批改 + 高级反馈）。
- **校级授权：** 按学校规模收取年度授权费，包含全校教师账号 + 数据看板 + 专属支持。
- **API 服务（远期）：** 向教育平台提供批改能力 API 接口，按调用量计费。

### 6. 关键资源（Key Resources）

- **AI 模型能力：** InternLM-Math（数学推理）、PaddleOCR（文字识别）、UniMERNet（公式识别）、YOLO（图像分割与目标检测）——均为可调用的开源模型或 API。
- **物理知识库：** 人教版高中物理教材知识图谱 + 标准答案库 + 解题模板 + 常见错误库，是系统批改准确性的核心壁垒。
- **开发团队：** 5 人团队，覆盖 CV、NLP、后端、前端、测试全栈能力。
- **计算资源：** 云端 API 调用为主（降低硬件门槛），辅以本地轻量模型。

### 7. 关键活动（Key Activities）

- **模型集成与调优：** 将 OCR、公式识别、大模型批改等能力串联成完整 Pipeline，持续优化 Prompt 和识别精度。
- **知识库建设：** 整理高中物理教材内容，构建结构化知识库并持续扩充，这是长期的内容运营活动。
- **平台开发与迭代：** 前后端开发、用户体验优化、功能迭代。
- **数据收集与标注：** 收集真实学生作业样本，标注训练数据，持续提升模型效果。

### 8. 关键合作伙伴（Key Partners）

- **模型提供方：** 上海 AI Lab（InternLM-Math）、百度（PaddleOCR）等开源模型社区。
- **教育内容方：** 高中物理教师（提供标准答案和常见错误案例）、教材出版社（知识体系授权）。
- **云服务商：** 阿里云 / 腾讯云（API 托管、向量数据库、对象存储）。
- **试点学校：** 合作学校提供真实使用场景和反馈数据。

### 9. 成本结构（Cost Structure）

- **API 调用成本（主要变动成本）：** 大模型推理（InternLM-Math）、OCR 识别、Embedding 计算，按调用量线性增长。
- **云基础设施：** 服务器、向量数据库、对象存储，月度固定支出。
- **开发人力：** 当前阶段为课程项目，无实际人力成本；产品化后为主要固定成本。
- **数据标注：** 物理作业样本收集和标注，一次性投入 + 持续少量维护。

## 二、MVP 功能列表

### 必须实现的 3 个核心功能

| # | 功能 | 理由 |
|---|------|------|
| 1 | **作业图像上传 + 预处理 + 文字/公式识别** | 这是整个系统的输入基础。没有可靠的图像识别，后续所有批改都无法进行。包含：拍照/文件上传 → 去噪/校正 → OCR 文字提取 + 公式转 LaTeX。 |
| 2 | **基于大模型的智能批改（文字 + 公式题）** | 核心价值所在。将识别出的学生答案与标准答案对比，利用 InternLM-Math 进行推导过程检查，输出评分 + 错误定位 + 改正建议。MVP 阶段聚焦力学计算题。 |
| 3 | **批改结果展示与反馈界面** | 用户直接感知到的价值交付。教师能看到每道题的评分、每一步的对错标注、具体错误原因和建议。无此界面，再强的后端也无法展示。 |

### 必须不去实现的 3 个功能

| # | 功能 | 排除理由 |
|---|------|----------|
| 1 | **手绘物理图形深度分析（受力图/电路图语义理解）** | 技术难度为五星（项目最大挑战），需要大量标注数据和专门模型训练。MVP 阶段将图形分析降级为「大模型直接识图给出参考意见」，不做精确的元素级语义解析。 |
| 2 | **多 Agent 智能体协作批改** | 架构复杂度高，涉及任务分解、Agent 间通信、结果整合。MVP 阶段用单一大模型 + RAG 的简单架构即可完成基础批改，待核心链路跑通后再引入 Agent 机制。 |
| 3 | **学生端长期学习画像与个性化推荐** | 需要长期数据积累才有意义，且涉及用户体系、数据持久化等额外工作量。MVP 阶段专注单次批改质量，不做跨时间维度的学情分析。 |
